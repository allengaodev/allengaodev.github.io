Module 1 - TLS/SSL Overview TLS/SSL Overview (00:40:02)

1.1 What is SSL? What is TLS? (06:57)

我們平常操作網站背後都是透過 ISP 提供的大量 Router 轉發封包來達成的。

一開始網路的操作因為只牽涉到讀取所以並不需要安全性，但是到後來的網路需要透過網路傳送一些隱私資料就發明了 SSL 與 TLS

SSL 與 TLS 可以用來保護網路傳輸是安全的、存取公司防火牆後的網路資源、用來建立 VPN 連線。

SSL（Secure Sockets Layer）是由 Netscape 在 1994 年開發的用來保護網際網路上的傳輸

TLS 是由 IETF 在 1999 年接手 SSL 之後重新改名而成的。

1.2 How do SSL/TLS Protect your Data? (04:37)

當 Client 傳資料給 Server 的這段過程很容易被線路中間的壞人捕獲，SSL/TLS 雖然是用是設計用來保護傳輸的過程，但是它並不能避免資料被捕獲。

SSL/TLS 用來保護資料的三個方式: Confidentiality: 確保資料只能被 Client 和 Server 讀懂內容，所以必須透過資料加密的方式確保資料被捕獲時壞人也看不懂。 Integrity: 防止資料傳輸的過程中被篡改，它並不能禁止壞人直接修改資料，它能確保資料被修改時 Server 方能查覺到。 Authentication: 確保 Client 和 Server 的來源是真的，兩台機器會透過類似驗證身分證的方式確保機器是真的機器。

Confidentiality 使用 Encryption 將資料加密。 Integrity 使用 Hashing 檢測資料是否被篡改。 Authentication 使用 PKI 驗證身分。

資料傳輸的過程還是會需要大量的公開 Router 來轉發，只不過有了 SSL/TLS 保護之後就算資料被捕獲也沒關係，因為有 Confidentiality， 也不用擔心資料的真假，因為有 Integrity，也不用擔心來源是否是真的機器，因為有 Authentication。

1.3 Anti-Replay and Non-Repudiation (04:32)

當銀行分支要傳送存款100元的封包給銀行總部時，中間的壞人就會捕獲這個封包並且複製多份相同內容後在傳給銀行總部，以此來欺騙總部相信有多個存款請求。

Anti-Replay 則是透過幫訊息加上序列編號來解決這個問題，這樣銀行總部就會將重複的封包內容捨棄掉，這個編號機制是透過 Integrity + Authentication 達成的。

Non-Repudiation 是代表發送訊息的一方不能否認訊息是它發出的，由於訊息有透過 Integrity 與 Authentication 進行保護， 也可以理解成傳送給對方的時候能保證沒有被篡改也確保是真人，那你就不能事後否認不是你本人發出這條訊息的。

1.4 Key Players (05:03)

SSL/TLS 生態系是由三個關鍵部分組成的:

  1. Client
  2. Server
  3. Certificate Authority(CA)

Client 代表發起 TLS 握手請求的一端，通常是指瀏覽器(Phone, Apps, IoT)，可以不用被 Authenticated，

Server 代表接收 TLS 握手請求的一端，通常是指 Web Server (IIS, Nginx, Load Balancer, SSl Accelerator)，通常必須要被 Authenticated。

當一個 Client 訪問 Bank.com 時， Bank.com 會回傳一份認證 (Certificate) 證明自己是真的網站並不是假冒的，這個動作就叫做 Authenticated， 要取得一份認證則需要先去 Certificate Authority 進行註冊，由於 CA 是具有公信力的，所以當我們的 Client 相信 CA 那麼 CA 相信的網站 Server 我們也應該要相信，最後結果就是 Client 相信 Server，這個流程叫做 Trust Anchor。

1.5 TLS / SSL Versions - Part 1 (05:48)

SSL 1.0 (1994) => SSL 2.0 (1995) => SSL 3.0 (1996) => TLS 1.0 (1999) => TLS 1.1 (2006) => TLS 1.2 (2008) => TLS 1.3 (2018)

現代的網站都使用 TLS 為主，有得時候語意上可以與 SSL 混用。

https://www.ssllabs.com/ssl-pulse/

2021/05月有 99.5% 網站支援 TLS 1.2 同時有 45.3% 支援 TLS 1.3 2024/11月有 99.9% 網站支援 TLS 1.2 同時有 70.1% 支援 TLS 1.3

首先要根據自己的網站需求來挑選支援的版本，如果網站需要支援很舊的設備那可能就需要支援舊版本的 TLS 1.0 或 TLS 1.1，如果只支援高版本的 TLS 雖然 Security 會增加但是會降低網站的 Accessibility。

2014/10月 SSL 3.0 網站支援率為 100% 但當月出現了 POODLE Attack，所以 2014/11月 SSL 3.0 網站支援率掉到剩下 60%。

1.6 TLS / SSL Versions - Part 2 (13:05)

SSL 1.0 (1994) => SSL 2.0 (1995) => SSL 3.0 (1996) => TLS 1.0 (1999) => TLS 1.1 (2006) => TLS 1.2 (2008) => TLS 1.3 (2018)

SSL 1.0 (1994)

由 Netscape 開發，沒有對外公開，被證實漏洞很多，不安全不建議使用。

SSL 2.0 (1995)

用 SSL 1.0 重新設計後推出，被證實漏洞很多，不安全不建議使用。

SSL 3.0 (1996)(RFC-6101)

用 SSL 2.0 重新設計後推出，TLS 都是從這個版本衍生出來的，新增了 Certificate Chains 與 Compression 功能，但是 Compression 功能被證實為不安全所以沒有人在使用，並且額外支援了更多 Key Exchange 功能例如 Diffie–Hellman key exchange， SSL 1.0 與 SSL 2.0 只支援 RSA key exchange。

最後 Netscape 將規格整理後推出 RFC-6101，從 1996 ~ 2014 年主要都是使用 SSL 3.0，因為出現了 POODLE Attack，現在 SSL 3.0 已經不安全不建議使用。 Major Version Minor Version SSL 1.0 1 0 SSL 2.0 2 0 SSL 3.0 3 0 TLS 1.0 3 1 TLS 1.1 3 2 TLS 1.2 3 3 TLS 1.3 3 4

雖然 TLS 1.3 被標註為小版本更新，但實際上改動很大。

TLS 1.0 (1999)(RFC-2246)

基本上就是 IETF 把 SSL 3.0 接管後把它改名成 TLS 1.0，並推出 RFC-2246 規格，功能上與 SSL 3.0 幾乎相同。 新增了 HMAC 還有必須支援 Additional Key Exchange

重大漏洞 BEAST Attack 攻擊 CBC 加密。

根據 2021/3月推出的 RF-8996，TSL 1.0 已經被列為不安全不建議使用

TLS 1.1 (2006)(RFC-4346)

功能上與 TLS 1.0 幾乎相同。 EXPORT grade 加密被正式棄用。 修復 CBC 攻擊漏洞 (BEAST Attack)。

根據 2021/3月推出的 RF-8996，TSL 1.1 已經被列為不安全不建議使用

TLS 1.2 (2008)(RFC-5246)

功能上與 TLS 1.1 幾乎相同。 加強 Key Generation 的安全性。 支援 Authenticated Encryption with Associated Data(AEAD) 加密。 安全建議使用。

TLS 1.3 (2018)(RFC-8446)

算是重大更新的版本。 Shorter Handshake(2 vs 5+) 0-RTT Resumption Forward Secrecy required AEAD Cipher required 並且移除了不安全的演算法，可能會影響兼容性。 安全建議使用。
